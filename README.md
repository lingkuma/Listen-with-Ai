# Listen-with-Ai

一个用于测试浏览器网页端 **麦克风**、**录屏**、**录屏音频轨** 能力的纯静态示例项目。

## 项目内容

本项目包含 3 个测试页面：

- `windows.html`：面向 Windows + Chrome / Edge
- `android.html`：面向 Android 浏览器
- `safari.html`：面向 Safari / iPhone / iPad

以及一个导航首页：

- `index.html`

## 测试目标

主要验证以下 3 类能力：

1. **麦克风获取能力**
   - 使用 `getUserMedia({ audio: true })`
   - 判断浏览器是否能正常申请并采集麦克风

2. **其他 App / 系统音频获取能力**
   - 使用 `getDisplayMedia({ video: true, audio: true })`
   - 观察录屏流里是否真的返回 `audio track`
   - 注意：这不代表网页可以越权抓任意 App 声音，而是测试浏览器在录屏授权下是否提供可用音频轨

3. **录屏能力**
   - 使用 `getDisplayMedia()` + `MediaRecorder`
   - 录制并回放最终结果，确认是否带有音频

## 结论预期

### Windows

- 麦克风：通常支持
- 录屏：通常支持
- 系统音频 / 标签页音频：**部分 Chromium 浏览器支持，但依赖共享方式和浏览器弹窗选项**

### Android

- 麦克风：通常支持
- 录屏：部分支持
- 其他 App 音频 / 系统音频：**网页端通常拿不到，很多情况下没有音频轨**

### Safari / iPhone / iPad

- 麦克风：通常支持
- 录屏：支持情况受系统版本影响较大
- 其他 App 音频 / 系统音频：**网页端基本不可得**

## 本地运行

由于媒体能力通常要求安全上下文，推荐使用以下方式启动：

### 方式 1：直接部署到 Vercel

这是最简单的方式。

1. 把项目推到 GitHub
2. 在 Vercel 中导入该仓库
3. 直接部署
4. 使用生成的 `https` 域名访问

本项目是纯静态页面，不需要额外构建配置。

### 方式 2：本地静态服务器

如果只是本机测试，可使用任意静态服务器，例如：

```bash
npx serve .
```

或：

```bash
npx http-server .
```

然后在浏览器中打开对应地址。

> 注意：部分浏览器在 `localhost` 下也允许媒体权限，但真机测试时仍建议使用 HTTPS。

## 使用步骤

1. 打开首页，进入对应平台测试页
2. 点击“检测浏览器能力”
3. 点击“申请麦克风”
4. 点击“申请录屏 + 音频”
5. 点击“开始录制已授权流”
6. 播放系统声音或页面声音
7. 停止录制并回放结果
8. 观察日志中的：
   - 是否支持 `getDisplayMedia`
   - `录屏流: audio=0` 还是 `audio>0`
   - 最终录制文件中是否真的有声音

## 重要说明

- **移动端网页通常无法获取其他 App 正在播放的声音**
- 即使你请求了 `audio: true`，浏览器也可能：
  - 直接忽略
  - 返回无音频轨的流
  - 只支持网页自身或标签页音频，而不是整个系统音频
- iOS 上所有浏览器都受 WebKit 能力边界影响，不能把 Chrome iOS 当成桌面 Chrome 看待

## 适合用途

- 浏览器媒体能力验证
- 跨平台兼容性测试
- 给产品 / 研发 / 测试团队做能力边界演示
- 录屏音频是否可用的快速验证
